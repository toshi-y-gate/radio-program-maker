## 1. プロジェクト概要

### 1.1 成果目標

トークスクリプトを入力するだけで、AIボイス（音声クローン含む）を使った高品質なラジオ番組を自動生成するWebアプリケーション。非エンジニアでも直感的に操作でき、将来的に商用サービス（サブスクリプション型）として最大10,000名規模に展開する。

### 1.2 成功指標

**定量的指標**:
- 1セリフの音声生成: 5秒以内（API応答時間依存）
- 10分番組の全体生成: 2分以内
- キャッシュによるAPI呼び出し削減率: 30〜60%
- 最大4名の同時話者対応

**定性的指標**:
- 非エンジニアでも迷わず操作できるUI
- プリセットボイスで手軽に高品質番組を作成できる体験
- BGM合成による本格的なラジオ番組の仕上がり

### 1.3 ターゲットユーザー

- ラジオ番組制作者・Podcast配信者
- 企業の社内放送・研修コンテンツ制作者
- 英会話スクール等の教育コンテンツ制作者
- 非エンジニア（直感的なWeb UIで操作可能）

## 3. ページ詳細仕様

### 3.1 Phase 1（MVP — Streamlit単一ページ構成）

#### P-001: 番組作成ページ（メインUI）

- **ページID**: P-001
- **目的**: スクリプト入力から音声生成・ダウンロードまでの一連の操作
- **ルート**: `/`（Streamlit デフォルト）

**機能詳細**:

| 要素 | 仕様 |
|------|------|
| スクリプト入力 | テキストエリア。対応フォーマット: `[話者名] セリフ` / `【話者名】 セリフ` / `話者名: セリフ` / `話者名： セリフ` |
| 話者検出 | スクリプトから話者を自動検出（最大4名） |
| ボイス割り当て | 各話者にプリセットボイスを選択（日本語4種・英語4種） |
| スクリプトプレビュー | 話者名＋セリフの一覧表示 |
| 音声生成 | MiniMax TTS APIでセリフごとに音声生成、進捗バー表示 |
| 再生プレビュー | ブラウザ上で結合済み音声を再生 |
| ダウンロード | MP3形式でダウンロード |

**サイドバー機能**:

| 要素 | 仕様 |
|------|------|
| API接続テスト | MiniMax APIへの接続確認ボタン |
| モデル選択 | speech-2.8-hd / speech-2.8-turbo / speech-2.6-hd / speech-2.6-turbo |
| 音声パラメータ | 話速（0.5〜2.0）、音量（0.1〜10.0）、ピッチ（-12〜+12）、感情（7種） |
| BGM設定 | ファイルアップロード（MP3/WAV/OGG）、挿入モード（5種）、音量・イントロ・アウトロ調整 |
| キャッシュ管理 | 件数・容量表示、クリアボタン |

**タブ構成**:
- 📝 番組作成: メイン操作エリア
- 📖 使い方: スクリプト書式ガイド

**処理フロー**:
```
スクリプト入力 → 解析・話者検出 → ボイス割り当て
    → [生成ボタン] → セリフごとにキャッシュ確認
        → HIT: キャッシュから取得
        → MISS: MiniMax API呼び出し → キャッシュ保存
    → 音声結合 → 音量正規化 → BGM合成（任意）
    → MP3出力 → 再生プレビュー & ダウンロード
```

## 4. データ設計概要

### 4.1 エンティティ（Phase 1）

| エンティティ | 説明 | 永続化方式 |
|-------------|------|-----------|
| ScriptLine | 話者名 + セリフテキスト | メモリ内（セッション中のみ） |
| AudioCache | 生成済み音声のキャッシュ | ファイルシステム（cache/） |
| BGMFile | アップロードされたBGM | ファイルシステム（bgm/） |
| OutputFile | 生成された番組音声 | ファイルシステム（output/） |

### 4.2 キャッシュキー構成

```
hash(text + voice_id + model + speed + volume + pitch + emotion) → MP3ファイル
```

### 4.3 バリデーション

| 対象 | ルール |
|------|--------|
| スクリプト | 空でないこと、少なくとも1つの話者が検出されること |
| 話者数 | 最大4名（超過時は先頭4名のみ使用） |
| BGMファイル | MP3/WAV/OGG形式のみ |
| APIキー | .envに設定済みであること |

## 5. セキュリティ要件

### 5.1 全フェーズ共通

- APIキーは `.env` で管理（Git管理外）
- APIキーのハードコーディング禁止
- 入力値サニタイゼーション（スクリプトテキスト）

### 5.2 将来対応（Phase 3以降）

- ヘルスチェックエンドポイント: `/api/health`
- グレースフルシャットダウン: SIGTERM対応、8秒タイムアウト
- HTTPS強制（本番環境）
- ブルートフォース対策（認証導入時）
- パスワードポリシー: 8文字以上
- セッション管理、CSRF対策

### アーキテクチャ

```
[ユーザー] → [Web UI (Streamlit)]
                    │
                    ├── スクリプト解析 (script_parser.py)
                    ├── キャッシュ確認 (cache.py)
```

## 10. 非機能要件

### パフォーマンス
- 1セリフの音声生成: 5秒以内（API応答時間依存）
- 10分番組の全体生成: 2分以内
- キャッシュヒット時: 即座に返却

### コスト管理
- キャッシュによるAPI呼び出し削減（目標: 30〜60%）
- Turbo/HDモデルの使い分けによるコスト最適化
- 将来: MiniMax Custom契約による単価交渉

### スケーラビリティ
- Phase 1: ローカル環境（単一ユーザー）
- Phase 3: マルチユーザー対応（〜1,000名）
- Phase 4: 商用スケール（〜10,000名）

## 11. リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| MiniMax API障害 | サービス停止 | フォールバック用のサブAPIを検討 |
| API料金値上げ | コスト増加 | キャッシュ強化、OSSモデル併用 |
| 音声品質の不満 | ユーザー離脱 | 複数モデル選択肢、ボイスクローン提供 |
| APIキー漏洩 | 不正利用 | 環境変数管理、キーローテーション |
| スケーリング限界 | 応答遅延 | キュー処理、非同期生成の導入 |
